## Твоя зона ответственности
Парсер расписания портала `rr-edu.ranepa.ru`. Это **чистая библиотека**: она
получает данные и возвращает их в виде моделей `ScheduleEvent`. В базу данных ты
НЕ пишешь — записью займётся sync-задача в боте, которая вызовет твой `fetch_schedule`,
а затем `db.upsert_schedule_events`.

## Что реализовать (`/parser/portal.py`), сигнатуры заморожены в §6:
```python
def authenticate(login: str, password: str) -> str
def fetch_schedule(token: str, since: date, until: date) -> list[ScheduleEvent]
```
- `authenticate` логинится на портал и возвращает **сессионный токен/cookie-строку**
  (НЕ пароль — пароль наружу не отдаём и нигде не сохраняем).
- `fetch_schedule` дёргает `GET /api/v1/private/scos/calendar` (и сопутствующие),
  подставляя токен, и маппит ответ в `list[ScheduleEvent]`. Каждое событие обязано
  иметь `external_id` (id из портала) — по нему БД делает идемпотентный upsert.

## ПЕРВАЯ задача — снять реальную авторизацию (без этого парсер не написать)
Механизм авторизации API ещё не зафиксирован. Сними его через DevTools:
1. Залогинься на портале → `F12` → вкладка **Network** → фильтр **Fetch/XHR**.
2. Открой страницу расписания, найди запрос к `/api/v1/private/scos/calendar`.
3. В **Request Headers** определи схему: `Authorization: Bearer …` (токен) либо `Cookie:` (сессия).
4. Проверь, был ли до него запрос логина (`/login`, `/token`, `/auth`), вернувший токен в JSON.
5. **Copy as cURL**, сохрани в `/parser/SAMPLE_AUTH.md`. ⚠️ Затри значение токена — храним только структуру заголовков, не секрет.
Сохрани также пример JSON-ответа календаря в `/parser/samples/calendar_response.json` — он нужен для офлайн-теста.
